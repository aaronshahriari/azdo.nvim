==============================================================================
azdo.nvim                                                            *azdo*

Work with Azure DevOps pull requests in Neovim. Wraps the Azure CLI (`az rest`
for the Azure DevOps REST API, plus local `git` for diffs) in a minimalist yet
effective workflow. A port of guh.nvim (GitHub) to Azure DevOps.

Tenets:
- Never lose data.
- Work with buffers and PR data over REST; the only thing needing a local
  checkout is the PR diff (Azure has no unified-diff endpoint).
- All mappings work in all Azdo buffers ("azdo://…").

==============================================================================
INSTALLATION                                                     *azdo-install*

Requirements:
- Nvim 0.13+
- Azure CLI ("az"), authenticated via `az login`: https://learn.microsoft.com/cli/azure/
- The "azure-devops" extension: `az extension add --name azure-devops`
- A local clone of the repo (for the PR diff).
- (Optional) For highlighting diffs, use a plugin such as: https://github.com/barrettruth/diffs.nvim

Auth uses your `az login` session; every REST call passes the Azure DevOps
`--resource` audience. No PAT is required for cloud. For on-prem Azure DevOps
Server (TFS) or PAT auth, see |azdo-config| (the `pat` and `base_url` options).

Install (lazy.nvim): >lua
  { 'https://github.com/<you>/azdo.nvim', opts = {} }
<
Install (vim.pack): >lua
  vim.pack.add{ 'https://github.com/<you>/azdo.nvim' }
  require('azdo').setup{}
<

CONFIGURATION                                                     *azdo-config*

Configure Azdo by calling its setup() function (lazy.nvim's `opts` does this
for you). All options are optional; the defaults target cloud Azure DevOps.
>lua
  require('azdo').setup({
    base_url          = nil,    -- on-prem collection root; nil = cloud
    api_version       = '7.1',  -- REST api-version
    pat               = nil,    -- PAT string, or a function returning one
    project           = nil,    -- default "org/project" for :Azdo items
    create_in_tab     = true,   -- open a new PR in a new tab
    pr                = { … },  -- PR-view settings (e.g. comments pane width %)
    items             = { … },  -- how `open_split` splits (direction + size)
    debug             = false,  -- log REST calls to stdpath('log')/azdo.log
    workitem_sections = { … },  -- editable work-item sections, per type
    keymaps           = { … },  -- buffer-local default keys, per action
    menu              = nil,    -- key for the :AzdoMenu palette, e.g. '<leader>a'
  })
<
                                                              *azdo-config-pat*
`pat`         (string|function|nil) Personal Access Token. When set, Azdo
              authenticates with the PAT (HTTP Basic) instead of `az login`. May
              be a function returning the token (e.g. to read it from a
              keychain). Also read from `$AZDO_PAT` / `$AZURE_DEVOPS_EXT_PAT`.
              Needs "Code (read & write)" scope, plus "Build (read)" for the
              CI-logs view. Required for on-prem Azure DevOps Server (TFS).

`base_url`    (string|nil) On-prem collection base URL, e.g.
              "https://tfs.example.com/tfs/MyCollection". When set, all REST
              urls are built from it instead of cloud "https://dev.azure.com".

`project`     (string|nil) Default project for the work-items dashboard, so it
              works without a local clone. |azdo-workitems|
              - Cloud:   "org/project"
              - On-prem: "project" (the collection is already in `base_url`)

`api_version` (string) REST API version. Default "7.1"; older on-prem servers
              may need e.g. "6.0".

`create_in_tab` (boolean) Open a newly-created PR in a new tab. Default true.

                                                          *azdo-config-pr*
`pr`          (table) PR-view settings (the `pr` / `prdiff` / `prcomments`
              buffers). |azdo-diff|
              - `comments.width` (number|nil) Width of the comments pane, as a
                percentage (1-99) of the editor width. nil keeps Vim's 50%
                default. E.g. `pr = { comments = { width = 30 } }` for a narrow
                comments pane.
              The legacy flat `comments_width` key still works (mapped onto
              `pr.comments.width`) but is deprecated.

                                                       *azdo-config-items*
`items`       (table) How the `open_split` mapping (|<Plug>(azdo-open-split)|,
              default `<C-W><CR>`) opens the PR / work item under the cursor —
              e.g. from the status or `:Azdo items` dashboard:
              - `split` ('vertical'|'horizontal') Split direction. Default
                'vertical' (side-by-side).
              - `size`  (number|nil) Split size, as a percentage (1-99) of the
                editor. For a vertical split this is the new window's width; for
                horizontal, its height. nil keeps Vim's default even split.
              - `sort`  (string) Default sort for the `:Azdo items` dashboard:
                'changed', 'created', 'id', 'title', 'type', 'state', or
                'assignee' (a trailing " date" is allowed, e.g. 'created date').
                Date sorts are newest-first; 'state' is in board order. Default
                'changed'. Re-sort interactively with the `s` keymap.
              - `group` (boolean) Group the dashboard under `###` subheadings
                derived from the sort (by day for dates, by state in board
                order, by type/assignee). 'id' and 'title' are never grouped.
                Default true; false renders a flat list. Toggle it from the `s`
                picker.
              - `fold`  (boolean|string[]) Fold groups closed by default
                (grouping only). false = all open (default), true = all closed,
                or a list of group labels (typically state names, e.g.
                `{ 'Done' }`) to start folded while the rest stay open — handy to
                collapse finished work without hiding it. Expand with the
                standard fold keys: `za` toggles the group under the cursor, `zR`
                opens all, `zM` closes all. The list applies on first render, so
                pair it with `sort = 'state'` (or whichever grouping produces
                those labels).
              - `assignee` ('me'|'all'|string|string[]) Who the dashboard
                lists. 'me' (default) = items assigned to you (`@Me`); 'all' =
                everyone's active items; a name string = one person (matched on
                display name or unique name); a list = several people (OR'd).
                Change it at runtime with the `assignee` keymap (default `ga`),
                which pops a multi-select of the project's active assignees.
              - `hide_states` (string[]) States to hide from the active list,
                case-insensitive, e.g. `{ 'Done' }`. Hidden items are filtered
                out before grouping and their count shows in the header; reveal
                them with the `toggle_hidden` keymap (default `gh`). The WIQL
                already drops Closed/Removed. Default `{}`. |azdo-workitems| >lua
    require('azdo').setup({
      items = {
        split = 'vertical',
        size = 40,
        sort = 'state',
        group = true,
        fold = true,
        hide_states = { 'Done' },
      },
    })
<
`debug`       (boolean) Log REST calls to `stdpath('log')/azdo.log`. Default
              false.

`menu`        (string|nil) If a key, maps it (normal mode) to the |:AzdoMenu|
              command palette. nil maps nothing.

                                                   *azdo-config-sections*
`workitem_sections`
              (table) Editable work-item sections, per type, for the work-item
              view and `c:` editor. |azdo-workitems| A
              `{ [type] = { {title, field}, … } }` table, where `field` is the
              Azure reference name; "default" applies to unlisted types.
              Overriding a type replaces that type's list (it is not spliced
              into the default), so include the base sections you still want: >lua
    require('azdo').setup({
      workitem_sections = {
        ["Product Backlog Item"] = {
          { "Description", "System.Description" },
          { "Acceptance Criteria", "Microsoft.VSTS.Common.AcceptanceCriteria" },
          -- Add org-specific custom fields by their reference name, e.g.:
          { "Technical Analysis", "MyOrg.Custom.TechnicalAnalysis" },
        },
      },
    })
<
                                                    *azdo-config-keymaps*
`keymaps`     (table|false) Buffer-local default mappings inside `azdo://`
              buffers, as `{ action = lhs }`. Override one action and the rest
              keep their defaults; set an action to `false` to drop just that
              key, or `keymaps = false` to disable all defaults (then map the
              `<Plug>(azdo-…)` maps yourself). The action names and default keys
              are listed under |azdo-mappings|. Example: >lua
    require('azdo').setup({
      keymaps = {
        merge = 'cm',   -- rebind "merge PR" from cM to cm
        logs  = false,  -- drop the CI-logs key
      },
    })
<
Example (on-prem, PAT auth): >lua
  require('azdo').setup({
    base_url = "https://tfs.example.com/tfs/MyCollection",
    pat      = os.getenv("AZDO_PAT"),
    project  = "MyProject",
  })
<

WORKFLOW                                                           *azdo-usage*

The `:Azdo*` commands and `<Plug>(azdo-…)` maps are registered at startup by
plugin/azdo.lua (|lua-plugin-init|); `require('azdo').setup{}` only applies your
configuration (|azdo-config|), so the commands work even before it runs.

                                                                     *azdo-api*
A Lua API mirrors the commands, for mapping directly or calling from your own
code: `require('azdo')` exposes `setup(opts)`, `status()`, `work_items()`,
`open(target)`, `menu()`, `create_pr()`, `merge()`, `review()`, `edit()`,
`link()`, `web()`, and `refresh()`.

To review a PR without doing a checkout in your repo:

- Run `:Azdo` to see your status (list of PRs, issues, etc.).
  - Hit <Enter> or `CTRL-W<Enter>` on a PR/issue to view it.
- Run `:Azdo [target]` to view a specific PR/issue. |azdo-commands|
  - Hit `R` to refresh the view.
- Run `:AzdoMenu` for a command palette if you'd rather not memorize the
  commands and mappings. |:AzdoMenu|
- All "azdo://…" buffers have the same basic mappings. |azdo-mappings|
- From the PR view, hit `dd` to toggle the diff. |azdo-diff|
- PR comments are loaded:
  - in a 'scrollbind' split window next to the diff window (if the
    scrollbind gets out of sync, try |:syncbind|).
  - as diagnostics on the diff buffer (severity INFO, source "azdo.nvim").
    Navigate via |]d|/|[d|; peek with |CTRL-W_d|.
  - in the quickfix list ("Azdo comments"). |]q| |[q| |:copen|
- Hit `cc` or `:AzdoComment` to comment on a diff line.

                                                                 *azdo-confirm*
To confirm an action (such as "merge", "post comment", "approve", etc.) which
accepts a (potentially optional) text input, Azdo presents a "azdo://…/edit".
- To confirm the action, just save-and-close the buffer: `:wq`, `ZZ`, etc.
- To cancel/abort the action, close-without-write (`:q!` or `ZQ`). Or if the
  action requires non-empty content, writing an empty buffer will also abort.

Keymaps                                                         *azdo-mappings*

Azdo defines global `<Plug>(azdo-…)` keys at startup. You can map anything to
those |<Plug>| keys to "override" the default Azdo keymaps. When a `azdo://`
buffer is opened it sets buffer-local default mappings unless you have defined
such an override.

Example: the `<Plug>(azdo-merge)` feature is mapped to "cM" by default, but you
can override it by mapping it to <F5>, then Azdo will not map "cM": >vim
  nmap <F5> <Plug>(azdo-merge)
<
Default mappings (local to `azdo://` buffers):

- Buffer-relative VIEW actions:
  - `R` Refresh the current `azdo://` buffer.
    <Plug>(azdo-refresh)
  - `dd` Toggle PR diff + comments. |azdo-diff|
    <Plug>(azdo-diff-toggle)
    (<Plug>(azdo-diff) opens/refreshes the diff without toggling it off.)
  - `dl` View the most-recent CI logs for all job-names in the CI matrix.
    <Plug>(azdo-logs)
  - `]f` View the next PR commit (from the most-recent push).
    <Plug>(azdo-next-commit)
  - `[f` View the previous PR commit (from the most-recent push).
    <Plug>(azdo-prev-commit)
  - `gw` Open this PR/work-item in the web browser.
    <Plug>(azdo-web)
  - `g?` View this help section.
    <Plug>(azdo-help)
- Buffer-relative UPDATE actions:
  - `cC` Comment on the PR/issue (top-level). |azdo-comment|
    <Plug>(azdo-comment-overview)
  - `cM` Merge PR. |azdo-merge|
    <Plug>(azdo-merge)
  - `cR` Review PR: approve / comment / request-changes (sets reviewer vote).
    <Plug>(azdo-review)
  - `c:` Edit PR title/description, or a work item's sections. |azdo-workitems|
    <Plug>(azdo-edit)
  - `gW` Link a work item assigned to you to this PR. |:AzdoLink|
    <Plug>(azdo-link)
- Cursor-relative actions:
  - `<Enter>` invokes |:Azdo| on the |WORD| at cursor.
    <Plug>(azdo-open)
  - `CTRL-W<Enter>` invokes |:Azdo| on the |WORD| at cursor, in a split window.
    <Plug>(azdo-open-split)
  - `cc` (or `c` in Visual mode): Comment on a diff-line. |azdo-comment|
    In a `azdo://…/prcomments/` buffer, edits the comment at cursor.
    <Plug>(azdo-comment)
  - `cr` (or `c` in Visual mode): Reply-to/Resolve a thread. |azdo-comment|
    In a `azdo://…/prcomments/` buffer, acts upon thread at cursor.
    <Plug>(azdo-comment)
  - `t` Tag/untag a work item (a local "pin"). |azdo-workitems|
    In the work-items dashboard, acts on the `#id` under the cursor; in a
    work-item view, acts on that item.
    <Plug>(azdo-tag-toggle)
  - `s` Sort the work-items dashboard. Prompts (via |vim.ui.select()|, so it
    flows through Telescope / fzf-lua / snacks / the default picker) for a
    field, then re-orders (and re-groups) the list in place. The picker also
    has an entry to toggle grouping. |azdo-workitems|
    <Plug>(azdo-sort)
  - `cc` Set the work item's State. Prompts (via |vim.ui.select()|) with the
    states valid for that item's type, then patches it — the dashboard's
    keyboard equivalent of dragging a card between board columns.
    In the dashboard, acts on the `#id` under the cursor; in a work-item view,
    on that item. |azdo-workitems|
    <Plug>(azdo-set-state)
  - `gh` Show/hide the dashboard's hidden states (`items.hide_states`). Only
    meaningful when some states are configured to hide. |azdo-workitems|
    <Plug>(azdo-toggle-hidden)
  - `ga` Filter the dashboard by assignee. Pops a multi-select popup (a markdown
    float) of "All assignees", "Me", and the project's active assignees; toggle
    entries with `<Space>`, apply with `<CR>`, cancel with `q`/`<Esc>`. Refetches
    with the new filter; the choice sticks across refreshes. |azdo-workitems|
    <Plug>(azdo-assignee)

View PR diff                                                        *azdo-diff*

`dd` on a PR view toggles the diff + comments in 'scrollbind' split windows:
press it to open them, press it again to close the comments split and return to
the PR overview. (<Plug>(azdo-diff) always opens/refreshes; <Plug>(azdo-diff-toggle),
bound to `dd`, toggles.)

- OUTDATED diffs (if any) are shown as extra diff section above the "current"
  diff, if any UNRESOLVED comments are anchored thereto.
- Unresolved comments on OUTDATED code (i.e. code from an old push), are
  attached to diff-hunks with filename format `outdated-<id>:<file>`.
- UNRESOLVED comments on code OUTSIDE of the PR diff, are similarly attached
  to diff-hunks with filename format `outside-<id>:<file>`.
- The current, latest diff follows those quasi-diffs, and is always shown.
- Only UNRESOLVED comments are shown.
- Note: Azure DevOps exposes no per-file "Viewed" state via its API, so the
  VIEWED-file collapsing below is effectively inert (nothing is collapsed).

PR diff:

- Lists diff-hunks in this order:
  1. Outdated (only if unresolved comments)
  2. Outside (only if unresolved comments)
  3. Current
- Shows VIEWED files as a single line (diff/comments not loaded): >
  (viewed) path/to/file
<
- The diff overlay shows: >
  PR diff (Viewed files: 3/12, Resolved comments: 4/20).
<
- Refresh ("R") to update after marking/unmarking a file as VIEWED.

PR comments:

- in a 'scrollbind' split window next to the diff window (if the scrollbind
  gets out of sync, try |:syncbind|).
- when multiple comments anchor to the same diff line, every comment's
  heading row is guaranteed; the remaining vertical budget is filled with
  comment bodies in-order. If a comment can't fit, it shows `(truncated)`. The
  full comment is always accessible via the diagnostic peek (`<C-w>d`) or the
  quickfix list.
- as diagnostics on the diff buffer (severity INFO, source "azdo.nvim"):
  - Navigate using `vim.diagnostic.jump` (|]d| |[d|).
  - See comment details using `<C-w>d` |CTRL-W_d-default|.
- in the quickfix list ("Azdo comments") via |vim.diagnostic.setqflist()|.
  Use |]q|/|[q| or |:copen| to browse.

Comment on issues, PRs, and diffs                                *azdo-comment*

`cc` on a PR diff-line, or on an existing comment, opens an editable
`azdo://…/comment/<id>` buffer. Write-and-close to post the comment. |azdo-confirm|

`cr` on a comment (thread), presents a menu:
1. Reply
2. Resolve

See |:AzdoComment|.

------------------------------------------------------------------------------
Merging PRs                                                        *azdo-merge*

`cM` on a PR view opens a picker:
- 1. squash
- 2. merge
- 3. rebase
- 4. squash --admin (bypass branch protections)
- 5. merge --admin (bypass branch protections)
- 6. rebase --admin (bypass branch protections)

`[count]cM` skips the picker (e.g. `1cM` = squash, `5cM` = merge --admin).

"merge" maps to the noFastForward strategy, "squash" to squash, "rebase" to
rebase. `--admin` sets `bypassPolicy` to complete past branch policies.

For "squash" and "merge", Azdo presents a `azdo://…/merge/<id>` buffer,
prefilled with the PR title and body (first line = subject, rest = body) used
as the merge-commit message. Write-and-close to complete the PR. |azdo-confirm|

------------------------------------------------------------------------------
Review PRs: approve, request changes, or reject                   *azdo-review*

`cR` on a PR view opens a picker ("approve"/"request-changes"/"comment").
`[count]cR` skips the picker:
- `1cR` = approve
- `2cR` = comment
- `3cR` = request-changes

To confirm the "review" action, Azdo presents a `azdo://…/review/<id>` buffer.
Write-and-close to confirm the "review" action. |azdo-confirm|

COMMANDS                                                        *azdo-commands*

:Azdo [target]                                                           *:Azdo*

Views or reloads the given [target], or shows a status if [target] is not
given (or is a repo). Accepts |<mods>|: >

    :vertical Azdo …
    :horizontal Azdo …
    :tab Azdo …

If |<mods>| is not given, the existing window for [target] (if any) is
focused.

Without [target]: shows a status dashboard listing the repo's active PRs
(for the current `azdo://` buffer's repo or the |current-directory|'s repo).
Hit <Enter> or `CTRL-W<Enter>` on a `#id` to view that PR. |azdo-mappings|

A numeric [target] is probed: an existing PR id opens the PR; otherwise it is
treated as a work-item id.

[target] "items" (or "mine" / "workitems") opens the work-items dashboard for
the items assigned to you. |azdo-workitems|

With [target]: views or refreshes the PR, work-item, commit, or repo-status:
- PR or work-item number (current repo): `:Azdo 42`
- Commit SHA (current repo): `:Azdo a1b2c3d`
- PR URL: `:Azdo https://dev.azure.com/org/project/_git/repo/pullrequest/42`
- Commit URL: `:Azdo https://dev.azure.com/org/project/_git/repo/commit/a1b2c3d`
- Work-item URL: `:Azdo https://dev.azure.com/org/project/_workitems/edit/42`
- Bare repo URL (status view): `:Azdo https://dev.azure.com/org/project/_git/repo`
- Azdo buffer name: `:Azdo azdo://org/project/repo/pr/42`
- Slug ("org/project/repo#42" or repo-only "org/project/repo") >
    :Azdo org/project/repo#42
    :Azdo #42
    :Azdo org/project/repo
<
For a PR target, `:Azdo` loads the pr/ + prdiff/ + prcomments/ views.
The pr/ view becomes the |alternate-file| of the prdiff/ view (so `CTRL-^`
returns to it).

Commits are shown as a |ft-gitcommit-plugin| buffer (via local `git show`).
<Enter> on a SHA in any `azdo://` buffer opens that commit.

:[range]AzdoComment                                               *:AzdoComment*

Creates, updates, or deletes a comment. >

    :.AzdoComment
    :'<,'>AzdoComment

Use range "%" to create a top-level comment on the current PR/issue: >

    :%AzdoComment

Use bang "!" with a single-line [range] to delete a comment: >

    :.AzdoComment!
<

:AzdoCreate                                                       *:AzdoCreate*

Creates a PR. Prompts for the source branch (defaults to the current branch)
and the target branch (defaults to the remote's HEAD branch, else "main") --
both with <Tab> completion over local + remote branches. Then opens an editor
with the title prefilled from the source branch's tip commit subject and an
empty description: first line = title, rest = description. ZZ creates the PR
(ZQ aborts). The source branch must already be pushed.

On success the new PR's diff view opens in a new tab; set
`create_in_tab = false` in setup() to open it in the current window instead.

:AzdoLink                                                           *:AzdoLink*

From a PR buffer, links a work item assigned to you to the current PR. Lists
your active assigned work items (WIQL `[System.AssignedTo] = @Me`) and links
the chosen one. Also bound to `gW`. |azdo-mappings|

:AzdoMenu                                                           *:AzdoMenu*

Opens a command palette: a |vim.ui.select()| launcher for azdo.nvim's top-level
entry points (status, work items, create PR, open-by-target), so you don't have
to remember command names. The same list is shown everywhere — it's a launcher,
not a per-buffer keymap list (use `g?` inside an `azdo://` buffer for that).

Because it routes through |vim.ui.select()|, whatever picker you have wired to
it (Telescope, fzf-lua, snacks, or the builtin) renders the palette — no extra
config needed. Bind it for quick access, e.g.: >lua

    vim.keymap.set('n', '<leader>a', '<Plug>(azdo-menu)')
<
                                                                  *azdo-menu-api*
For a custom picker (e.g. a Telescope previewer that shows each action's blurb)
the action list behind `:AzdoMenu` is exposed as a small module: >lua

    local menu = require('azdo.menu')
    menu.available()   --> the palette actions, each a { id, label, desc, run }
    menu.exec(action)  --  run one
<

WORK ITEMS                                                     *azdo-workitems*

Dashboard ~

`:Azdo items` opens a dashboard of the work items assigned to you (`items`,
`mine`, and `workitems` are all accepted). By default it runs the WIQL query
`[System.AssignedTo] = @Me`, active (not Closed/Removed), newest-changed first;
the `ga` keymap (below) re-targets it at "all" or specific people.
|<mods>| work, e.g. `:vertical Azdo items`.

The dashboard needs only a project, not a repo. It is resolved from, in order:
the `project` option, the current `azdo://` buffer's repo, then the |current-
directory|'s clone. Set `project` (|azdo-config|) to run `:Azdo items` anywhere.

Each dashboard line shows `#id  [type/state]  title`. When the list spans more
than one person (e.g. the "all" filter), each line also carries the assignee as
a styled `` `@Name` `` token; a single-assignee list (such as your own items)
omits it, since it would just repeat.

Hit <Enter> (or `CTRL-W<Enter>`) on a `#id` to open that work item. The view
is rendered from the REST API (`pat` / `base_url` aware) as
markdown: title, metadata, and one `##` section per editable field for the
work-item type (HTML converted to markdown; empty fields show "(empty)").
`gw` opens it in the browser.

Editing ~

Put the cursor in a section (under its `## <header>`, or the title area) and hit
`c:`. |<Plug>(azdo-edit)| That opens an editable, multi-line buffer for just
that one field, prefilled with its current value as markdown. Edit, then `ZZ`
to save (`ZQ` to abort), exactly like comment/PR compose. |azdo-confirm|

On save the field is converted markdown -> HTML (the title is saved as plain
text) and PATCHed on its own; every other field is left untouched, so existing
rich formatting elsewhere on the item is preserved.

The sections shown depend on the work-item type. Defaults:
- Product Backlog Item / Task / etc.: Description, Acceptance Criteria.
- Bug: Repro Steps, System Info, Acceptance Criteria.
Add org-specific custom fields per type with `workitem_sections`. |azdo-config-sections|

Tagging (pinning) ~

Press `t` to tag/untag a work item — on the `#id` under the cursor in the
dashboard, or on the item in a work-item view. |<Plug>(azdo-tag-toggle)|

Tagged items appear in a "★ Tagged" section at the top of the dashboard. A
tagged item is fetched by id, so it shows even when it is closed or assigned
to someone else. Tags are local, persisted across sessions, and keyed per
project (so each project shows its own pins):
>
    stdpath('data')/azdo/tags.json
<
Filtering by assignee ~

Press `ga` in the dashboard to choose whose items it lists.
|<Plug>(azdo-assignee)| A small markdown popup appears with "All assignees",
"Me", and a roster of people. Toggle entries with `<Space>` (or `<Tab>`), apply
with `<CR>`, cancel with `q`/`<Esc>`. "All assignees" is mutually exclusive with
the individual picks. Applying refetches the dashboard, since the assignee is
part of the WIQL query; the choice sticks across `R` refreshes.

The roster comes from `items.assignees` when you set it — a fixed list of the
people you filter by, shown instantly with no extra query (recommended): >lua
    require('azdo').setup({
      items = { assignees = { 'Aaron Shahriari', 'Jane Doe', 'John Roe' } },
    })
<
When `items.assignees` is unset, the roster is instead derived from the project's
most-recently-active work items (one bounded query, `$top` 200) — a little slower
on a large project, and it only sees who's been active lately.

Set the default audience (who the dashboard lists on open) with `items.assignee`
in |azdo-config|: >lua
    require('azdo').setup({ items = { assignee = 'me' } })  -- default
    require('azdo').setup({ items = { assignee = 'all' } }) -- everyone
    require('azdo').setup({ items = { assignee = 'Jane Doe' } })
    require('azdo').setup({ items = { assignee = { 'Jane Doe', 'John Roe' } } })
<
'me' uses `@Me` (robust to display-name quirks); a name string or list is matched
in WIQL against each person's display name or unique name (e.g. their email).

Sorting ~

Press `s` in the dashboard to re-order the list. It prompts (via
|vim.ui.select()|, so a Telescope / fzf-lua / snacks UI is used if installed)
for a field — changed date, created date, id, title, type, state, or assignee —
and re-renders both sections in place. The choice sticks across `R` refreshes.
|<Plug>(azdo-sort)|

Set the default sort with `items.sort` in |azdo-config|. A trailing " date" is
accepted, so all of these are equivalent: >lua
    require('azdo').setup({ items = { sort = 'created' } })
    require('azdo').setup({ items = { sort = 'created date' } })
<
Date sorts are newest-first; the default is 'changed' (most-recently-updated
first), matching the order the server returns. The 'state' sort uses board
order (New/Approved → Committed → In Progress → Resolved → Testing → Done),
not alphabetical.

Grouping ~

By default the list is grouped under `###` subheadings derived from the sort:
by day for date sorts (e.g. "June 27, 2026"), by state in board order (like the
columns of the Azure board), and by type or assignee for those sorts. 'id' and
'title' are never grouped. Toggle grouping from the `s` picker, or set the
default with `items.group` in |azdo-config|.

Groups are folds. Set `items.fold = true` to have them all start closed, or a
list of group labels to start only those folded — e.g. `items.fold = { 'Done' }`
with `sort = 'state'` collapses the Done column while the rest stay open (a way
to tuck finished work away without hiding it like `hide_states` does). Then use
the standard fold keys to expand: `za` toggles the group under the cursor, `zR`
opens all, `zM` closes all. A closed group's fold line shows its item count.

Hiding ~

Set `items.hide_states` (e.g. `{ 'Done' }`) to drop those states from the
active list — handy for tucking away finished work. Hidden items are filtered
before grouping and their count shows in the header ("… · 3 hidden"); the WIQL
already drops Closed/Removed. Press `gh` to reveal them (and again to re-hide).
|<Plug>(azdo-toggle-hidden)|

States ~

Press `cc` on a work item — the `#id` under the cursor in the dashboard, or the
item in a work-item view — to change its State. |<Plug>(azdo-set-state)| It
prompts (via |vim.ui.select()|) with the states valid for that item's type (a
Bug offers different states than a Product Backlog Item), then patches
`System.State`. The dashboard updates in place; a later `R` pulls fresh. This
is the keyboard equivalent of dragging a card between columns on the board.

    require('azdo').setup({ debug = true })

    :log azdo
